DNS FUNDAMENTALS

What is DNS?

DNS (Domain Name System) converts human-friendly domain names into machine-friendly IP addresses. Without DNS, the internet as we know it couldn't exist for humans.

REAL-WORLD ANALOGY

DNS is like your phone's contact list. You don't remember phone numbers — you save Mom → 9876543210. Similarly, DNS maps google.com → 142.250.x.x. Without it, you'd memorize a 12-digit IP for every website you visit.

🧠

HUMAN-READABLE

google.com
Easy to remember, type, and share. Used in browsers, emails, and URLs. Meaningless to network hardware.
🖥️

MACHINE-READABLE

142.250.183.14
Used by routers and network infrastructure. Impossible to memorize for thousands of sites. Required for actual routing.

KEY INSIGHT

DNS happens BEFORE HTTP. Before any website loads, before any data transfers, DNS silently finds the server's IP address. No DNS → No IP → No HTTP → No website.

📍
APPLICATION LAYER
DNS operates at the Application Layer of the TCP/IP model — same layer as HTTP, FTP, SMTP.
📡
UDP PORT 53
DNS uses UDP port 53 for standard queries (fast). TCP port 53 for large responses or zone transfers.
⚡
MILLISECONDS
The entire DNS resolution process typically completes in under 100ms — invisible to the user.
💾
TTL CACHING
DNS responses are cached (TTL = Time To Live). Repeat visits skip DNS lookup, improving speed.

CYBERSECURITY PERSPECTIVE

DNS as an Attacker's Goldmine

Before touching the target, attackers read its DNS records. DNS is publicly accessible — no login, no hacking, just querying. It reveals organizational infrastructure silently.

ATTACKER MINDSET

"DNS reveals the company's internal structure without touching the system. One domain lookup can reveal mail servers, cloud providers, third-party services, APIs, and hidden dev environments."

🎯
A RECORD RECON
Find the real server IP. Use for Nmap scanning, identifying hosting provider (AWS/Azure), and bypassing CDN protection.
dig example.com
📧
MX RECORD RECON
Identify mail servers. Used for phishing preparation, spear phishing campaigns, and discovering email infrastructure weaknesses.
dig mx example.com
🔑
TXT RECORD LEAKAGE
TXT records reveal SPF policies, domain verifications, and third-party services (Google Workspace, SendGrid, etc.) — exposing the tech stack.
dig txt example.com
🌿
SUBDOMAIN DISCOVERY
dev.example.com, admin.example.com, staging.example.com — subdomains are often less secured than the main site. Each is a potential attack entry point.
sublist3r -d example.com
💥
ZONE TRANSFER (AXFR)
Misconfigured DNS servers can dump their entire record set to anyone who asks. This reveals ALL subdomains, internal servers, mail systems in one request.
dig axfr @ns1.example.com example.com
🔄
REVERSE DNS (PTR)
Map IP addresses back to domain names. When scanning IP ranges with Nmap, PTR records reveal which domains live on each IP — expanding the target list.
dig -x <ip-address>
PENTEST WORKFLOW

Every web pentest begins with DNS recon. One domain becomes 20+ subdomains + mail servers + API servers + dev servers. If you master DNS, you already understand half of the recon phase.

DHCP FUNDAMENTALS

What is DHCP?

DHCP (Dynamic Host Configuration Protocol) automatically assigns network configuration to devices when they join a network. Without DHCP, every device would need manual IP configuration.

REAL-WORLD ANALOGY

DHCP is like hotel reception. When you check in, the receptionist automatically assigns you a room number (IP), gives you the floor plan (subnet mask), tells you the main entrance (gateway), and the concierge number (DNS). You don't configure anything — it just works.

WITHOUT DHCP
✗ Admin must manually configure every device
✗ IP conflicts when two devices use same address
✗ Scaling to 1000 devices is a nightmare
✗ Mobile devices can't auto-join new networks
✗ No guest Wi-Fi possible
WITH DHCP
✓ Devices auto-configure in seconds
✓ Conflict prevention via IP pool management
✓ Scale to thousands of devices effortlessly
✓ Phones, laptops, IoT join seamlessly
✓ Guest networks are trivially managed
DHCP COMPONENTS
🖥️
DHCP SERVER
Manages IP pool and assigns addresses with configuration options. Usually your router or a dedicated server.
💻
DHCP CLIENT
Any device requesting IP config — PC, phone, printer, IoT sensor. Sends DISCOVER to start the process.
🔀
DHCP RELAY
Forwards DHCP messages between subnets when client and server are on different networks.
🏊
IP POOL
Range of available IP addresses. Server allocates from this pool. Exclusions/reservations can be set.
⏱️
LEASE
Time period the assigned IP is valid. Client must renew before expiry. Prevents IP hoarding.
🌐
DHCP OPTIONS
Extra config delivered with IP: default gateway, DNS servers, domain name, NTP servers.
DHCP PORTS

DHCP uses UDP (connectionless — fast and simple for broadcast communication):

UDP 67 → DHCP Server UDP 68 → DHCP Client

DORA is the 4-step conversation between a DHCP client and server. Every time a device joins a network, this exchange happens automatically in milliseconds.

Click "ANIMATE DORA" to watch the IP assignment process step by step...
D — DISCOVER
Client broadcasts to the entire network: "Is there a DHCP server here? I need an IP address!" Sent as broadcast because client has no IP yet and doesn't know the server's address.
Source: 0.0.0.0 → Dest: 255.255.255.255
O — OFFER
DHCP server responds with an available IP address offer, including subnet mask, gateway, DNS, and lease time. Server reserves this IP temporarily while waiting for the client's response.
Offered IP + Config Options → Client MAC
R — REQUEST
Client formally requests the offered IP. This is still broadcast to notify ALL DHCP servers which offer was accepted — so other servers release their reserved addresses.
Broadcast: "Accepting offer from server X"
A — ACKNOWLEDGE
Server confirms the IP assignment and sends full configuration. Client now has a valid IP address, subnet mask, default gateway, and DNS server. Network communication begins.
IP Lease + Full Config → Client is online ✓
SECURITY THREATS

DHCP Attacks & Defenses

DHCP has no built-in authentication. Any device on the LAN can send DHCP messages. This creates serious security risks that attackers actively exploit.

🌊
DHCP STARVATION
Attacker floods the network with fake DHCP requests using spoofed MAC addresses, exhausting the entire IP pool. Legitimate devices can no longer get IPs → Denial of Service.
DoSIP ExhaustionMAC Spoofing
👿
ROGUE DHCP SERVER
Attacker sets up a fake DHCP server that responds faster than the legitimate one. Victims receive attacker-controlled gateway and DNS settings, redirecting all traffic.
MITMTraffic Interception
🕵️
MAN-IN-THE-MIDDLE
Via rogue DHCP, attacker pushes their machine as the default gateway. All network traffic flows through the attacker who can read, modify, or drop packets silently.
Passive MITMSSL Strip
🎣
DNS MANIPULATION
Rogue DHCP pushes attacker-controlled DNS server addresses. Victims' DNS queries go to attacker's server, enabling domain hijacking and phishing site redirection.
DNS HijackPhishing
DEFENSIVE MEASURES
🛡️
DHCP SNOOPING
Switch feature that allows DHCP replies only on trusted ports. Untrusted ports (user devices) can only send DHCP requests, not replies — blocking rogue servers.
🔒
PORT SECURITY
Limit MAC addresses per switch port. Rate-limit DHCP requests per port to prevent starvation attacks by spoofed MAC floods.
📊
MONITORING & LOGGING
Log all DHCP lease events (assign/renew/release). Detect anomalies like sudden surge in DISCOVER messages or unknown DHCP servers appearing on the network.
🔐
IP/MAC BINDING
Reserve specific IPs for known MAC addresses. Filter unknown MACs. Makes it harder for rogue clients to obtain valid IP configurations.
SMB FUNDAMENTALS

Server Message Block (SMB)

SMB is the protocol that enables file sharing, printer sharing, and network resource communication in Windows environments. It's the backbone of enterprise Windows networking.

WHAT SMB DOES
  • File sharing across the network
  • Printer sharing between computers
  • Windows domain authentication
  • Inter-process communication (IPC)
  • Active Directory communication
🏢
REAL-WORLD USE CASES
  • Office shared drives (\\fileserver\docs)
  • Network printers in enterprises
  • Active Directory domain environments
  • Windows file servers
  • NAS (Network Attached Storage)
⚠️
SMB SECURITY RISKS
  • EternalBlue (CVE-2017-0144) — WannaCry
  • Null session anonymous access
  • SMB relay attacks (NTLM capture)
  • Weak share permissions exposure
  • Outdated SMBv1 protocol still enabled
SMB VERSIONS
SMBv1Deprecated — EternalBlue target. Disable!
SMBv2Windows Vista+ improved performance
SMBv3Windows 8+ with encryption support
ATTACKER PERSPECTIVE

Open SMB port (445) is one of the most scanned ports in the world. EternalBlue allowed WannaCry to spread to 200,000+ machines globally via SMBv1. Always disable SMBv1.

PROTOCOLS WORKING TOGETHER

The Complete Network Lifecycle

DNS, DHCP, and SMB don't work in isolation. Here's what happens from the moment a device joins a network to when it opens a file on a server — every protocol plays its role.

01
DHCP — DORA
Device joins network → DHCP assigns IP, subnet, gateway, DNS server
02
DNS — RESOLUTION
Browser queries DNS to convert domain → IP address (before HTTP)
03
TCP/IP — HANDSHAKE
3-way handshake (SYN → SYN-ACK → ACK) establishes connection to server IP
04
HTTP/HTTPS — COMMUNICATION
Browser sends HTTP request → Server returns web page content
05
SMB — FILE ACCESS
User connects to \\fileserver\share → SMB authenticates and provides file access
